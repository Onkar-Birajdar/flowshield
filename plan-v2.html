Section 01
System Boundaries — Two Modules, One Repo
Addressing mentor feedback: "FlowShield is mixed with job execution. That's messy." This is the fix.

The mentor's critique was correct: the previous doc blurred FlowShield (the gateway/protection layer) with the job processing system (execution, workers, scheduling). They solve different problems, they scale differently, and they fail differently. They must be conceptually separate — even if they live in one repository.

The rule: FlowShield module owns everything that happens before a job is accepted into the system. The Processor module owns everything that happens after. The boundary is the database write of a new QUEUED job.

🛡 Module 1
flowshield-gateway

The API surface. Everything that protects the system from abuse and guarantees correctness at the entry point.

  • Tenant authentication (API key → tenant resolution)
  • IP-based throttling (per-IP rate window)
  • Per-tenant token bucket rate limiting (Redis Lua)
  • Idempotency enforcement (Redis SETNX + DB constraint)
  • Request validation (payload schema, required fields)
  • Abuse detection (burst scoring, suspicious patterns)
  • Job submission API (POST /api/v1/jobs)
  • Job status polling (GET /api/v1/jobs/{id})
  • Admin API (tenant management, quota config)
  • Publishes to SQS queue on acceptance
Explicitly NOT in this module
  • Job execution logic
  • Worker thread management
  • Retry / backoff logic
  • Watchdog / recovery
  • Priority scheduling
⚙ Module 2
flowshield-processor

The execution engine. Consumes from SQS, runs jobs, handles failures, ensures at-least-once delivery.

  • Consumes from SQS queue
  • Writes job to Postgres (status=QUEUED)
  • Worker pool (SELECT FOR UPDATE SKIP LOCKED)
  • Job handler registry (pluggable handlers)
  • Retry with exponential backoff + jitter
  • Visibility timeout + watchdog recovery
  • Priority-weighted scheduling with aging
  • Worker heartbeat (extends lock on long jobs)
  • Dead-letter queue (after max retries)
  • Job event audit trail
Why separate deployment is valuable
  • Gateway and Processor scale independently
  • Gateway can be deployed to multiple AZs
  • Processor outage doesn't affect job acceptance
  • Different resource profiles: Gateway=CPU, Processor=I/O
One repo, two deployable JARs. The monorepo has a Gradle multi-project build: flowshield-gateway and flowshield-processor each produce independent Spring Boot fat JARs. They share flowshield-common (domain models, DTOs, constants). Each deploys to its own EC2 instance. This is the cleanest architecture for a portfolio project — clean enough to be impressive, simple enough to actually build.

The boundary is SQS. The Gateway writes to Postgres (job record) and publishes to SQS (trigger). The Processor consumes from SQS and takes ownership. This means: Gateway going down doesn't lose jobs that were already queued. Processor going down doesn't affect new job acceptance (SQS buffers). These are independent failure domains.

Low-Level Design — Component Detail
Every class, every interface, every interaction that matters. This is what you'll be asked about in interviews.

Gateway Module — Component Breakdown

filter
IpThrottleFilter
Extracts IP from X-Forwarded-For header (handles proxies)
Redis: INCR ip:throttle:{ip} + EXPIRE 60 (fixed window/min)
Threshold 1: >200/min → 429 immediately
Threshold 2: >500/min → publish to abuse:events Redis stream
FAIL_OPEN: if Redis unavailable, log and pass through
Depends: RedisTemplate · AbuseEventPublisher
filter
TenantAuthFilter
Reads X-API-Key header → SHA-256 hash
L1: Redis GET tenant:{keyHash} (5min TTL, stores TenantContext JSON)
L2: DB query on cache miss → populate cache
Sets TenantContext in request attribute for downstream filters
401 on missing key. 403 on inactive tenant.
Depends: RedisTemplate · TenantRepository
filter
AbuseDetectionFilter
Reads abuse:score:{tenantId} from Redis (0-100)
Score >80: block with 403 "Tenant suspended for abuse"
Score 50-80: override plan → FREE tier limits temporarily
Score decays 5 pts/hour (implemented via TTL + initial value)
Listens to abuse:events stream to increment scores async
Depends: RedisTemplate · AbuseScoreService
filter
RateLimitFilter
Loads plan limits from TenantContext (set by AuthFilter)
Executes token_bucket.lua via EVALSHA (cached SHA on startup)
ALLOW: sets X-RateLimit-Remaining header, continues
DENY: 429 + Retry-After + X-RateLimit-Reset headers
Micrometer timer: measures Lua round-trip latency per call
Circuit breaker: FAIL_OPEN for bulk jobs, FAIL_CLOSED for payments
Depends: LettuceConnectionFactory · MeterRegistry
filter
IdempotencyFilter
Requires X-Idempotency-Key header (400 if missing on POST)
Redis SETNX idem:{tenantId}:{key} EX 86400 → stores job_id
Cache hit: fetch existing job from DB, return 200 immediately
DB UNIQUE constraint on (tenant_id, idempotency_key) as backstop
On constraint violation: convert to 200 with existing job (no 409)
Depends: RedisTemplate · JobRepository
controller
JobController
POST /api/v1/jobs → validates payload via Bean Validation
Inherits tenant from TenantContext (set by AuthFilter)
Inserts job to Postgres → publishes to SQS → returns 202 Accepted
GET /api/v1/jobs/{id} → tenant-scoped status query (can't see other tenants' jobs)
POST /api/v1/jobs/{id}/cancel → sets status=CANCELLED if QUEUED
Depends: JobRepository · SqsPublisher · TenantContext

Processor Module — Component Breakdown

consumer
SqsJobConsumer
@SqsListener("flowshield-jobs") — Spring Cloud AWS
Receives job_id from SQS message body (not full payload)
Fetches full job from Postgres (SQS = trigger, DB = source of truth)
Idempotent: if job already PROCESSING/DONE, ACK and skip
On DB error: don't ACK → SQS redelivers after visibility timeout
Depends: JobRepository · WorkerOrchestrator
worker
WorkerOrchestrator
ThreadPoolExecutor — size from flowshield.workers.count (default 5)
Worker poll loop: JPQL with FOR UPDATE SKIP LOCKED
Claims job: sets locked_by, locked_until = now()+30s
Resolves handler via JobHandlerRegistry.getHandler(job.type)
On graceful shutdown: shuttingDown=true, drain current jobs, max 60s wait
Depends: JobRepository · JobHandlerRegistry · HeartbeatService
interface
JobHandler
void execute(Job job) throws JobExecutionException
Implementations: SendEmailHandler, ProcessPaymentHandler, GenerateReportHandler
Contract: handlers MUST be idempotent (at-least-once delivery)
@Component + @JobType("SEND_EMAIL") annotation for auto-registration
Checked exceptions: RetryableException vs FatalException (skip retry)
Implementations are in consumer bounded context
scheduler
WatchdogScheduler
@Scheduled(fixedDelay=10000) — runs every 10 seconds
Recovery: UPDATE jobs WHERE locked_until < now() AND locked_by IS NOT NULL
CAS pattern: only updates if both conditions still true (race-safe)
Priority aging: decrement priority every 60s for long-waiting jobs (floor=1)
Alerts Micrometer counter on each recovery (paging signal)
Depends: JobRepository · MeterRegistry
service
HeartbeatService
Runs in separate thread per active worker
Every 20s: UPDATE jobs SET locked_until=now()+30s WHERE id=? AND locked_by=?
CAS: checks locked_by — prevents race with watchdog during slow jobs
Stops automatically when worker marks job DONE/FAILED
Solves: report jobs running 2-4 min without getting re-queued
Depends: JobRepository · ScheduledExecutorService
service
RetryDecisionService
On failure: determines if exception is RetryableException
Calculates: visible_after = now() + clamp(2^n × 1000, 1000, 30000) + rand(0,1000) ms
After max_attempts: marks DEAD, writes DEAD event, never retries
FatalException (e.g., invalid payload): skip to DEAD immediately
Logs structured error with all context for post-mortem
Depends: JobRepository · JobEventRepository

// ── GATEWAY REQUEST LIFECYCLE (single POST /api/v1/jobs) ─────────────────── //

  Client Request → Spring Filter Chain
                                                              Typical latency
  ① IpThrottleFilter
     └─► Redis INCR + EXPIRE                                  ~0.3ms
     └─► If >200/min → 429 (stop here)

  ② TenantAuthFilter
     └─► Redis GET tenant:{hash}  (cache hit ~95%)            ~0.3ms
     └─► Postgres SELECT  (cache miss ~5%)                    ~3–5ms
     └─► If not found → 401 (stop here)

  ③ AbuseDetectionFilter
     └─► Redis GET abuse:score:{tenantId}                     ~0.2ms
     └─► If score >80 → 403 (stop here)

  ④ RateLimitFilter
     └─► Redis EVALSHA token_bucket.lua                       ~0.4ms
     └─► If deny → 429 + Retry-After (stop here)

  ⑤ IdempotencyFilter
     └─► Redis SETNX idem:{tenantId}:{key}                    ~0.3ms
     └─► If key existed → 200 (stop here, no DB hit)

  ⑥ JobController
     └─► Validate payload (Bean Validation, in-memory)        ~0.1ms
     └─► Postgres INSERT jobs                                  ~3–5ms
     └─► SQS SendMessage                                      ~15ms  ← this is the only "slow" step
     └─► Return 202 Accepted

  Total P99 (happy path, cache hot): ~20ms
  Total P99 (cache miss): ~25ms
  SQS async option: fire-and-forget → reduces to ~5ms but loses delivery guarantee

Tech Stack — Every Choice Justified
Not "I used X because it's popular." Every choice has a tradeoff rationale you can defend in an interview.

Gateway Language
Java 17
Spring Boot 3.2
 Your strongest language. Virtual Threads (Project Loom) in Spring Boot 3.2 give near-reactive throughput with blocking code. Spring's filter chain is the right abstraction for multi-step request processing. Go (faster cold start, but you'd be learning language + architecture simultaneously), Node.js (single-threaded, rate limiting at high concurrency is harder) JVM startup time (~3s vs Go's ~100ms). Mitigated by keeping instances always warm on EC2.
Rate Limit Store
Redis 7
Lua Scripts
Lettuce
 Sub-millisecond latency for the hot path. Lua scripts execute atomically on Redis server — no TOCTOU race. Lettuce is non-blocking (important for filter chain throughput). Redis TIME in Lua eliminates clock drift. In-memory (fails with multiple instances), Postgres (5–10ms per check, adds latency), Hazelcast (operational overhead) Redis is an additional infrastructure component. Mitigated by FAIL_OPEN on Redis outage for non-critical paths.
Primary Database
PostgreSQL 16
FOR UPDATE SKIP LOCKED
Flyway
 FOR UPDATE SKIP LOCKED is native Postgres — no external queue broker needed. JSONB for flexible job payloads. UNIQUE constraint for idempotency. Partial index on QUEUED rows keeps worker pickup fast as table grows. MySQL (SKIP LOCKED available but Postgres JSONB is superior), MongoDB (no SKIP LOCKED equivalent, harder to enforce consistency) Postgres at very high write load (~5K inserts/sec) needs connection pooling (PgBouncer) + read replicas. Acceptable for current scale.
Message Queue
AWS SQS (FIFO)
 Free tier: 1M requests/month. Zero operational overhead (fully managed). Spring Cloud AWS integration is 20 lines of config. Exactly-once processing on FIFO queues with message deduplication IDs. Survives Gateway restart with no message loss. Kafka (operational overhead not justified at this scale — Kafka shines at >100K msgs/sec or when replay is needed), RabbitMQ (adds another managed service), Redis LPUSH/BRPOP (loses durability on Redis restart) SQS has ~15ms publish latency vs Kafka's ~5ms. Acceptable. SQS max message size: 256KB — job payloads must stay under this (enforced by validation).
Frontend
React 18
Vite
TailwindCSS
Recharts
TanStack Query
 React you specified. Vite for fast builds (10× faster than CRA). TanStack Query for server-state management (auto-refetch for live metrics). Recharts for time-series traffic charts. WebSocket via SockJS+STOMP for real-time security events. Next.js (SSR not needed for an admin dashboard), Vue (preference decision), Angular (too heavy for a portfolio dashboard) Client-side only routing (no SSR) means no SEO. Irrelevant for an admin tool.
Frontend Hosting
S3 + CloudFront
 S3 static hosting: $0.023/GB/month. CloudFront CDN: free tier 1TB/month data transfer. Custom domain via Route 53. HTTPS via ACM (free). This is how real companies host SPAs. "Deployed to S3 + CloudFront" is a stronger resume signal than Vercel/Netlify. Vercel/Netlify (easier but doesn't add AWS signal), Nginx on EC2 (unnecessary complexity for static files) CloudFront cache invalidation needed on each deploy ($0.005/1000 paths — negligible).
Testing
Testcontainers

RestAssured
k6
 Testcontainers: real Postgres + Redis + LocalStack (SQS) in Docker. No mocking infrastructure. RestAssured for readable integration tests. k6 for load testing with real throughput numbers. H2 in-memory (doesn't test SKIP LOCKED, JSONB, partial indexes), Mockito-only (misses integration bugs) Testcontainers tests are slower (~15s startup). Run in a separate CI stage from unit tests.
CI/CD
GitHub Actions
AWS CodeDeploy
 GitHub Actions: free for public repos. CodeDeploy: free, integrates natively with EC2. Pipeline: push → test → build JAR → push to S3 → CodeDeploy agent pulls and restarts. Zero-downtime via rolling deploy. Jenkins (self-hosted overhead), CircleCI (costs money), AWS CodePipeline (more complex setup for same result) CodeDeploy requires CodeDeploy agent on EC2 instances (one-time 10min setup).
Observability
Micrometer

 Micrometer: Spring's native metrics abstraction. CloudWatch: free tier 10 custom metrics. Spring Actuator for health checks (used by ALB health checks). No Grafana/Prometheus infrastructure needed — CloudWatch handles everything on AWS free tier. Prometheus + Grafana (beautiful but adds 2 more services to manage on EC2), Datadog (costs money) CloudWatch dashboards are less beautiful than Grafana. Acceptable for a portfolio project.

-- ── TENANT PLANS: business rules encoded as data ─────────────────────────── --
CREATE TABLE tenant_plans (
  plan                    VARCHAR(20)  PRIMARY KEY,           -- FREE | PRO | ENTERPRISE
  max_jobs_per_minute     INT          NOT NULL,             -- token bucket capacity + rate
  max_concurrent_jobs     INT          NOT NULL,             -- max PROCESSING state
  max_queue_depth         INT          NOT NULL,             -- max QUEUED state
  max_retries             INT          NOT NULL DEFAULT 3,
  base_priority           INT          NOT NULL DEFAULT 5,    -- 1=highest: ENT=1, PRO=5, FREE=10
  sla_critical_seconds    INT          NOT NULL DEFAULT 30,   -- P99 target for CRITICAL jobs
  fail_open_on_rate_limit BOOLEAN      NOT NULL DEFAULT false  -- allow through if Redis down?
);
INSERT INTO tenant_plans VALUES
  ('FREE',       10,   2,   100,  3, 10, 300, true),
  ('PRO',       100,  20,  1000, 5,  5, 60,  false),
  ('ENTERPRISE',1000, 200, 10000,10, 1, 30,  false);

-- ── TENANTS ───────────────────────────────────────────────────────────────── --
CREATE TABLE tenants (
  id              UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  name            VARCHAR(255) NOT NULL,
  api_key_hash    CHAR(64)     UNIQUE NOT NULL,      -- SHA-256 hex. NEVER store raw key.
  plan            VARCHAR(20)  NOT NULL REFERENCES tenant_plans(plan),
  is_active       BOOLEAN      NOT NULL DEFAULT true,
  abuse_score     INT          NOT NULL DEFAULT 0 CHECK (abuse_score BETWEEN 0 AND 100),
  created_at      TIMESTAMPTZ  NOT NULL DEFAULT now(),
  last_active_at  TIMESTAMPTZ
);

-- ── JOBS: the core entity ─────────────────────────────────────────────────── --
CREATE TABLE jobs (
  id               UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id        UUID         NOT NULL REFERENCES tenants(id),
  idempotency_key  VARCHAR(255) NOT NULL,
  type             VARCHAR(100) NOT NULL,
  priority         INT          NOT NULL DEFAULT 5,         -- 1=highest; from plan at submission
  job_priority_class VARCHAR(10) NOT NULL DEFAULT 'NORMAL',  -- CRITICAL|NORMAL|BULK
  payload          JSONB        NOT NULL,
  status           VARCHAR(20)  NOT NULL DEFAULT 'QUEUED',  -- QUEUED|PROCESSING|DONE|FAILED|DEAD|CANCELLED
  attempt_count    INT          NOT NULL DEFAULT 0,
  max_attempts     INT          NOT NULL DEFAULT 3,
  visible_after    TIMESTAMPTZ  NOT NULL DEFAULT now(),     -- don't pick up before this (backoff)
  locked_by        VARCHAR(100),                             -- worker_id (UUID per instance)
  locked_until     TIMESTAMPTZ,                              -- visibility timeout
  last_error       TEXT,
  created_at       TIMESTAMPTZ  NOT NULL DEFAULT now(),
  completed_at     TIMESTAMPTZ,
  source_ip        INET,                                     -- for abuse analysis

  CONSTRAINT uq_job_idempotency UNIQUE (tenant_id, idempotency_key)
);

-- Worker pickup: the most critical query in the system
-- Partial index: only QUEUED rows. Stays small as completed jobs grow.
CREATE INDEX idx_jobs_worker_pickup
  ON jobs (priority ASC, visible_after ASC)
  WHERE status = 'QUEUED';

-- Watchdog recovery: finds stale locked jobs
CREATE INDEX idx_jobs_watchdog
  ON jobs (locked_until)
  WHERE status = 'QUEUED' AND locked_by IS NOT NULL;

-- Tenant status queries: GET /api/v1/jobs?status=QUEUED
CREATE INDEX idx_jobs_tenant_status ON jobs (tenant_id, status, created_at DESC);

-- ── JOB EVENTS: immutable audit trail ────────────────────────────────────── --
CREATE TABLE job_events (
  id          BIGSERIAL    PRIMARY KEY,
  job_id      UUID         NOT NULL REFERENCES jobs(id),
  event_type  VARCHAR(30)  NOT NULL,  -- ENQUEUED|PICKED_UP|HEARTBEAT|RETRY|FAILED|COMPLETED|DEAD
  worker_id   VARCHAR(100),
  attempt     INT,
  message     TEXT,
  created_at  TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE INDEX idx_job_events_job_id ON job_events (job_id, created_at);

-- ── SECURITY EVENTS: abuse tracking ─────────────────────────────────────── --
CREATE TABLE security_events (
  id          BIGSERIAL    PRIMARY KEY,
  tenant_id   UUID         REFERENCES tenants(id),  -- nullable (unknown tenant on auth failure)
  event_type  VARCHAR(50)  NOT NULL,  -- IP_THROTTLE|AUTH_FAILURE|RATE_LIMIT|ABUSE_BLOCK|IDEMPOTENCY_DUPLICATE
  source_ip   INET,
  details     JSONB,
  created_at  TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE INDEX idx_security_events_tenant ON security_events (tenant_id, created_at DESC);
CREATE INDEX idx_security_events_ip ON security_events (source_ip, created_at DESC);

// ── AUTHENTICATION: all non-admin endpoints require X-API-Key header ──────── //

// ─── JOB SUBMISSION ──────────────────────────────────────────────────────── //

POST   /api/v1/jobs
  Headers: X-API-Key (required), X-Idempotency-Key (required), X-Forwarded-For (set by proxy)
  Body: { type: string, priority_class: "CRITICAL"|"NORMAL"|"BULK", payload: object }
  202 Accepted:    { job_id, status: "QUEUED", created_at, retry_after: null }
  200 OK:          { job_id, status: "...", created_at }  ← idempotency hit (existing job)
  400 Bad Request: { error: "VALIDATION_ERROR", fields: [{field, message}] }
  401 Unauthorized:{ error: "INVALID_API_KEY" }
  403 Forbidden:   { error: "TENANT_SUSPENDED", reason: "abuse_score_exceeded" }
  429 Too Many:    { error: "RATE_LIMITED", retry_after_seconds: 12 }
    Response headers on 202: X-RateLimit-Remaining, X-RateLimit-Reset
    Response headers on 429: Retry-After, X-RateLimit-Remaining: 0

GET    /api/v1/jobs/{id}
  Headers: X-API-Key (required) — tenant-scoped: can only see own jobs
  200 OK:    { job_id, type, status, attempt_count, created_at, completed_at, last_error }
  403:       returned if job belongs to a different tenant (not 404 — prevents enumeration)
  404:       { error: "JOB_NOT_FOUND" }

POST   /api/v1/jobs/{id}/cancel
  200 OK:    { job_id, status: "CANCELLED" }  ← only if job is QUEUED
  409:       { error: "CANNOT_CANCEL", reason: "job_already_processing" }

GET    /api/v1/jobs
  Query: ?status=QUEUED|PROCESSING|DONE|FAILED|DEAD&type=SEND_EMAIL&page=0&size=20
  200 OK:    { items: [Job], total: int, page: int, has_more: bool }

// ─── QUOTA INFO ───────────────────────────────────────────────────────────── //

Scale Analysis — What Breaks at 10K RPS and How to Fix It
Missing from previous doc. This is what interviewers ask immediately after you describe the design. Have these numbers ready.

The current design targets current scale: ~100–500 RPS. This section models what happens at 10× (1K RPS) and 100× (10K RPS), identifies the bottleneck at each tier, and specifies the fix. This is how a senior engineer thinks — not just "it works" but "here's where it breaks and what I'd do."

Redis — Rate Limiter
Token Bucket Throughput
Redis ops capacity: ~100K ops/sec (single node)
At 500 RPS: 500 Lua/s = 0.5% capacity ✓
At 1K RPS: 1K Lua/s = 1% capacity ✓
At 10K RPS: 10K Lua/s = 10% capacity ✓
At 100K RPS: 100K Lua/s = 100% → bottleneck ⚠
Fix at 100K RPS: Redis Cluster with consistent hashing on tenant_id. Each shard handles ~33K ops/sec. Zero application code change — Lettuce Cluster handles routing transparently. Add this as a design note; don't build it now.
PostgreSQL — Job Writes
INSERT Throughput
Postgres write capacity: ~3–5K INSERTs/sec (t3.medium RDS, SSD)
At 500 RPS: 500 INSERTs/sec = 10–17% ✓
At 1K RPS: 1K INSERTs/sec = 20–33% ✓
At 5K RPS: 5K INSERTs/sec → approaching limit ⚠
At 10K RPS: 10K INSERTs/sec → overloaded ✗
Fix at 5K+ RPS: (1) PgBouncer connection pooling — reduces connection overhead by 80%. (2) Batch inserts: buffer 100ms of job submissions, INSERT in batches (100× throughput). (3) Partitioning by created_at (monthly) — keeps active partition small. (4) Write-optimised instance (db.r5.large).
PostgreSQL — Worker Pickup
SKIP LOCKED Contention
5 workers × 1 query/200ms = 25 queries/sec
On partial index (QUEUED only): ~2ms P99 ✓
At 100K total jobs, 10% QUEUED: index = 10K rows ✓
At 10 instances × 20 workers: 200 concurrent skips → row contention ⚠
Fix at high worker concurrency: Add LIMIT 1 strictly to worker query (already done). Partition jobs by (tenant_id % N) and assign workers to partitions — eliminates cross-worker contention. Add more worker instances; SKIP LOCKED scales linearly with no coordination overhead.
SQS — Message Pipeline
Publish + Consume Latency
SQS FIFO: 3K msg/sec per message group
At 1K RPS: 1K msg/sec ✓
At 3K+ RPS with 1 message group: limit ⚠
Fix: hash tenant_id into 10 message groups → 30K RPS ✓
SQS publish latency: ~15ms average. This is the dominant latency in the gateway happy path. If this is unacceptable, switch SQS to async fire-and-forget (gateway doesn't wait for SQS ACK). Tradeoff: risk of accepted job not reaching processor if SQS publish fails after DB write. Handled by a background reconciler that republishes jobs stuck in QUEUED with no SQS evidence.

Failure Mode Analysis — 6 Deep Dives
"You touched failures, but not deeply enough." Every scenario: what triggers it, what breaks, how the system responds, how you recover.
⚡
FM-1: Redis Partial Outage (Primary down, replica available)
Trigger: AWS ElastiCache primary node failure. Failover in progress (~60s).
P1
What breaks
Rate limit checks fail (Lua script executes on primary). Idempotency cache fails. Tenant auth cache fails. IP throttling fails. During failover window: all 4 Redis-dependent filters are unavailable.
System response
Circuit breaker detects Redis unavailable within 50ms. Each filter has a configured fallback: IpThrottle → FAIL_OPEN (log, pass through). TenantAuth → FAIL_OPEN (fall back to DB). RateLimit → FAIL_OPEN for non-payment, FAIL_CLOSED (503) for payment jobs. Idempotency → DB UNIQUE constraint only. System degrades gracefully — non-payment job submission works, payment submission returns 503.
Recovery
ElastiCache promotes replica in ~60s. Circuit breaker detects recovery. All filters resume Redis operations. Token bucket keys are rebuilt on first request per tenant — no data migration needed (bucket state is ephemeral). Alert: redis_connection_failures > 5 fires PagerDuty within 30s of outage.
🐘
FM-2: Postgres Slow (Disk I/O saturation → query timeouts)
Trigger: RDS I/O burst credit exhausted. All DB queries taking >5s.
P0
What breaks
Job INSERT times out → Gateway returns 503. Worker pickup queries time out → workers idle (no new jobs claimed). Watchdog queries time out → stale locks not recovered. Queue depth grows unboundedly in SQS (processor can't consume fast enough).
System response
Spring Boot HikariCP connection pool: max connection wait = 3s → returns 503 Service Unavailable to clients rather than hanging. Workers enter exponential backoff on DB connection failures — poll every 5s instead of every 200ms, reducing DB load. SQS message visibility timeout ensures no SQS messages are lost (they become visible again after 30s and are reprocessed once DB recovers).
Recovery + Prevention
Immediate fix: Scale up RDS instance (vertical). Longer-term fix: Add PgBouncer to reduce connection overhead. Enable RDS Storage Autoscaling. Alert: rds_database_connections = max_connections * 0.8 fires P1 before saturation. Add read replica for status-query endpoints (GET /jobs).
🔄
FM-3: Worker Crash Loop (Handler throwing uncaught exceptions repeatedly)
Trigger: Bad deploy introduces a null pointer in SendEmailHandler. All SEND_EMAIL jobs crash workers.
P1
What breaks
Workers keep picking up SEND_EMAIL jobs, crashing immediately, releasing the lock. Watchdog sees the jobs as stale (locked_until expired) and re-queues them. Jobs cycle QUEUED → crash → QUEUED endlessly. attempt_count increments each time — eventually reaches max_attempts and jobs go DEAD.
System response
RetryDecisionService: detects NullPointerException as a FatalException → marks job DEAD immediately, skips retry. Prevents the endless cycle. Micrometer counter: flowshield_dead_jobs_total{type="SEND_EMAIL"} spikes → CloudWatch alarm fires P1 within 2 minutes. Other job types (PROCESS_PAYMENT) unaffected — workers skip SEND_EMAIL type jobs.
Recovery
Fix the handler bug → deploy → DEAD jobs can be requeued via admin API (POST /api/v1/admin/jobs/requeue?status=DEAD&type=SEND_EMAIL). This resets attempt_count=0, status=QUEUED, visible_after=now() for all affected jobs. Zero data loss — jobs are in DB, just in DEAD state.
🌐
FM-4: Network Partition (Gateway ↔ Processor network split)
Trigger: AWS AZ-level network issue. Gateway in AZ-1a, Processor in AZ-1b.
P1
What breaks
SQS publish from Gateway may fail (SQS is regional but network path is disrupted). Jobs are written to Postgres (QUEUED) but SQS message never sent. Processor doesn't know about these jobs — they sit in QUEUED forever, never picked up.
System response
The reconciler safety net: A scheduled job in the Processor (every 5 minutes) queries: SELECT id FROM jobs WHERE status='QUEUED' AND created_at < now()-interval '5 minutes' AND locked_by IS NULL. These are "orphaned" jobs — written to DB but never triggered via SQS. The reconciler republishes their job_ids to SQS. This is the defense against Gateway→SQS failures of any cause.
Recovery
Network partition resolves. Reconciler runs and republishes all orphaned jobs within 5 minutes. Jobs execute with a max delay of 5 minutes — acceptable for non-real-time job types. SDE-2 design principle: "Don't rely on SQS for correctness. Use it for performance. The DB is the source of truth. Any job in the DB will eventually execute."
💣
FM-5: Tenant Burst Attack (Coordinated DDoS from a single API key)
Trigger: Rogue tenant scripts 10,000 concurrent requests/second from 50 IPs.
P1
What breaks
Rate limiter caps tenant at their plan limit (e.g., 1000/min for ENTERPRISE). The remaining 9,000+ requests/sec still hit the filter chain, executing TenantAuth filter (Redis lookup) and IP throttle check. At 10K RPS, this is 10K Redis reads/sec from attack traffic alone. Redis CPU spikes. Legitimate tenants start seeing higher Redis latency.
System response
Layer 1 — IP throttle fires first: 50 IPs each hitting 200 req/min → all 50 IPs blocked within 60s. Subsequent requests from those IPs rejected at filter ①, never reaching auth or rate limit. Layer 2 — Abuse score: >500 req/min from any IP → abuse:score:{tenantId} += 10. Score reaches 80 within 8 minutes → tenant blocked at filter ③. Layer 3 — Manual: Admin blocks tenant via API.
Recovery
Tenant suspended. Admin investigates (security_events table has full IP history). If legitimate surge: re-enable with temporary limit increase. If malicious: revoke API key, issue new key on request. SLA impact for other tenants during attack: <2ms increase in P99 latency (IP throttle is a single Redis INCR — fast even under load).
🔁
FM-6: SQS Unavailability (Regional SQS outage)
Trigger: AWS SQS service disruption. Gateway can write to DB but not publish to SQS.
P2
What breaks
Gateway's JobController.publish() throws SqsException after writing to DB. Job is in QUEUED state in Postgres but Processor has no trigger to process it. If Gateway returns 503: client retries, idempotency key prevents duplicate DB entry. Job is orphaned until reconciler runs.
System response
Decision: Gateway catches SqsException, logs it with job_id, and still returns 202 Accepted to the client. The DB write is the commitment. Job will be processed — just delayed by up to 5 minutes until the reconciler picks it up. Client does not see a failure. Alternative (rejected): Return 503 on SQS failure. Worse UX, same outcome — client retries and idempotency prevents duplicate. The 202 approach is more honest: we accepted your job, it will be processed.
Recovery
SQS recovers. Reconciler (runs every 5min) finds orphaned QUEUED jobs, republishes to SQS. All jobs processed with <5min delay. Alert: sqs_publish_errors > 10 in 60s → P2 alert. Not P0 because jobs are safe in DB and will self-heal.

Security Layer — IP Throttling + Abuse Detection
"No IP-based limiting, no bot detection — huge gap." Here's the implementation.

The security layer is not a single component — it's a set of checks distributed across the filter chain, each catching a different threat model. The key principle: checks that require no DB access run first. Expensive checks (DB lookups) only run if cheap checks pass.

S1
IP-Based Throttling — First Line of Defense
CORE
Implementation
Redis fixed-window counter per IP. Key: ip:throttle:{ip}:{window} where window = floor(unix_ts / 60).

Threshold 1 (soft): >100 req/min → add to watchlist (ip:watchlist:{ip}, TTL 10min)
Threshold 2 (hard): >200 req/min → 429 immediately. No further processing.
Threshold 3 (abuse): >500 req/min → 429 + increment tenant abuse score + write security_event

IP extraction: reads X-Forwarded-For header, takes the last IP in the chain (the one added by our load balancer — not spoofable by clients). Falls back to request.getRemoteAddr().
Why This Threshold Math
A legitimate developer testing their integration will send <100 req/min. A retry loop bug might send 100–200 req/min (soft block, investigate). A DDoS script typically sends 500+ req/min from each IP. The thresholds are tuned to catch abuse while not blocking legitimate heavy users.

IP vs Tenant rate limiting: IP limiting protects against unauthenticated abuse (pre-auth attacks). Tenant rate limiting protects against authenticated overuse. Both are necessary. A rogue tenant using 50 different IPs bypasses per-IP limits — that's why tenant rate limiting at the next layer is essential.
S2
API Key Abuse Detection — Scoring System
CORE
Implementation
abuse:score:{tenantId} stored in Redis as an integer 0–100.

Score increments:
+5 for each rate-limit hit (above plan limit)
+10 for IP flagged as high-volume
+15 for auth failures with same API key
+20 for sending malformed payloads repeatedly

Score thresholds:
0–49: Normal operation
50–79: Override plan → FREE tier limits temporarily
80–100: Block all requests → 403

Score decay: TTL-based. Score key set with TTL = score × 60 seconds. A score of 80 decays fully in 80 minutes if no new incidents.
Why Scoring vs Hard Block
Hard blocking on first offense causes false positives. A misconfigured client hitting the rate limit 10 times shouldn't be blocked — they should be throttled to FREE tier until they fix their code. A genuine attacker who persists gets blocked.

Security events feed: Every score increment writes to security_events table (tenant_id, event_type, source_ip, details). The React dashboard shows this as a live feed. Operators can see the pattern — "same tenant hit rate limit 50 times from 5 IPs in 10 minutes" — and decide to manually block or contact the tenant.

What this signals to an interviewer: You've thought about the operational side of security. Not just "block the bad guys" but "give operators visibility to understand what's happening and make informed decisions."
S3
Additional Hardening Measures
EXTRA
Request Fingerprinting
Bot detection signal: Legitimate API clients typically have consistent User-Agent headers, consistent header ordering, and do not send unusual headers. A Redis HSET stores: fingerprint:{tenantId} → {user_agent, header_set_hash, first_seen}. When a tenant's request fingerprint changes dramatically (different language SDK suddenly, no User-Agent, unusual headers) → abuse score +5. Not a hard block — a signal.

Payload anomaly: Job payloads are validated against their type's JSON Schema. A payload consistently failing validation (different field names, wrong types) suggests automated tooling probing the API → score +3 per failure. After 5 failures: 400 with "Repeated validation failure — contact support".
Manual Block + Allowlist
Admin API: POST /api/v1/admin/security/block-ip → sets ip:blocked:{ip} = 1 in Redis with 24h TTL + writes to security_events.

IpThrottleFilter checks ip:blocked:{ip} before any other check. 403 immediately if present.

POST /api/v1/admin/security/allowlist-tenant → sets tenant:allowlisted:{tenantId} = 1. Allowlisted tenants bypass IP throttle and abuse detection (but still subject to plan rate limits). Use case: trusted partner running batch migrations.

Note on scope: Geo-blocking and threat intelligence (mentor's "Full" option) are left for v2. These require external APIs (MaxMind GeoIP, AbuseIPDB) and add ~2 weeks of work with marginal benefit for a portfolio project. Document as "v2 enhancements" in README.

FS-P3-01
SqsJobConsumer + WorkerOrchestrator
FEATURE

  • Concurrent test (Testcontainers): 5 workers, 5 jobs → each worker claims exactly 1. No duplicate claims.
FS-P3-02
Retry + Backoff + Dead-Letter + HeartbeatService
FEATURE
  • On failure: visible_after = now() + clamp(2^attempt × 1000, 1000, 30000) + random(0,1000) ms.
  • FatalException → skip to DEAD immediately (no retries). RetryableException → normal backoff.
  • HeartbeatService: per-worker thread, every 20s: UPDATE jobs SET locked_until=now()+30s WHERE id=? AND locked_by=?. CAS prevents heartbeat extending lock after watchdog recovery.
  • Test: handler that throws RetryableException 3 times → job reaches DEAD after max_attempts. Handler that throws FatalException once → DEAD immediately.
FS-P3-03
WatchdogScheduler + ReconcilerScheduler + Graceful Shutdown
INFRA
  • Watchdog (10s): CAS UPDATE recovers stale locks. Priority aging: decrement priority every 60s for long-waiting jobs.
  • Reconciler (5min): finds QUEUED jobs older than 5min with no SQS evidence → republishes to SQS. Guards against FM-4 (network partition) and FM-6 (SQS outage).
  • Graceful shutdown: @PreDestroy sets shuttingDown=true. Workers finish current job. Wait up to 60s. Then JVM exits.
  • Crash test: Kill JVM mid-execution → restart → watchdog recovers within 10s → job re-executes.
